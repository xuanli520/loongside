# Qwen

Use this when your team already runs on Qwen and you want that vendor family to stay explicit in Loong config instead of disappearing behind a generic compatibility route.

## At A Glance

| Field | Value |
| --- | --- |
| Built-in kind | `qwen` |
| Provider group | Direct Hosted Providers |
| Protocol family | `openai_chat_completions` |
| Feature family | `openai_compatible` |
| Auth scheme | `bearer` |
| Credential envs | DASHSCOPE_API_KEY, QWEN_API_KEY |
| Aliases | `qwen_compatible`, `dashscope` |
| Default base URL | `https://dashscope.aliyuncs.com` |
| Request endpoint | `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions` |
| Models endpoint | `https://dashscope.aliyuncs.com/compatible-mode/v1/models` |

## Minimal Config

```toml
active_provider = "qwen"

[providers.qwen]
kind = "qwen"
api_key = { env = "DASHSCOPE_API_KEY" }
model = "auto"
```

## Verify It

```bash
loong doctor
loong list-models
loong ask --message "Say hello and name the active provider."
```

If `list-models` is unreliable for this account or region, pin an explicit `provider.model` or add `preferred_models` instead of leaving recovery implicit.

## Auth And Routing Contract

| Contract | Value |
| --- | --- |
| Auth optional | no |
| Model probe auth optional | no |
| Default API key env | `DASHSCOPE_API_KEY` |
| OAuth env | none |
| Primary request route | `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions` |
| Primary model-catalog route | `https://dashscope.aliyuncs.com/compatible-mode/v1/models` |
